# Распознавание еды 2.0 — план от Codex (16.07.2026)

Сначала сверю актуальные возможности и лицензии подходящих food/vision-моделей и рантаймов, чтобы план был пригоден именно для Railway Linux с 8 GB RAM. Затем дам короткий приоритетный roadmap с оценкой трудозатрат.

Главная проблема не столько в поиске, сколько в постановке: сходство с одной каталожной картинкой нельзя показывать как вероятность блюда. `84%` у «Болоньезе» — это некалиброванный cosine score, а не 84% уверенности.

Приоритетный план:

1. **Сначала сделать честный benchmark — 1–2 дня**

- Собрать 200–500 реальных сканов с подтверждёнными ответами.
- Разделять train/test по пользователям и карточкам, чтобы одинаковые фото не утекали.
- Метрики: Recall@1, Recall@5, MRR и доля корректных отказов.
- Сравнить текущий CLIP с `SigLIP2-base-patch16-256`. SigLIP2 поддерживает мультиязычный image-text retrieval и подходит для названий на русском; base-модель достаточно компактна для 8 GB RAM. [Документация](https://huggingface.co/docs/transformers/model_doc/siglip2), [модель](https://huggingface.co/google/siglip2-base-patch16-256).
- Инференс: отдельный Python/ONNX-сервис, FP16/INT8, прогретый процесс. Не менять модель без offline A/B.

Ожидаю, что SigLIP2 даст заметный прирост, но сама замена энкодера проблему плохого индекса не решит.

2. **Перестроить индекс вокруг реальных фото — 2–4 дня**

Для каждой карточки хранить несколько прототипов:

- реальные фото из пользовательских отзывов — максимальный вес;
- подтверждённые сканы;
- реальные легально полученные меню-фото;
- каталожное/стоковое фото — малый вес;
- AI-generated фото исключить из retrieval либо оставить только как слабый fallback.

Не усреднять сразу все изображения. Считать:

```text
imageScore(card) =
  0.65 * max cosine(query, realPhoto)
+ 0.25 * mean(top-3 realPhoto similarities)
+ 0.10 * max cosine(query, catalogPhoto)
```

Одна плохая картинка тогда не испортит центроид, а несколько реальных ракурсов помогут против света, упаковки и подачи.

3. **Добавить image + text fusion — 2–3 дня**

Название и описание карточки кодировать **text-tower той же модели**, с несколькими русскими шаблонами:

- «фотография блюда том-ям паста»;
- «тарелка пасты с соусом том-ям»;
- название + категория + ключевые ингредиенты;
- синонимы: «клубника / свежая клубника / ягоды клубники».

Начальная формула:

```text
score =
  0.55 * realImageScore
+ 0.30 * textScore
+ 0.10 * categoryScore
+ 0.05 * popularityPrior
```

Если реальных фото у карточки нет — увеличить вес текста. Для этикеток отдельно сохранить нынешний OCR/OpenFoodFacts pipeline: визуальная модель не должна конкурировать с точным совпадением бренда.

4. **Использовать feedback сразу, без обучения большой модели — 2–4 дня**

Каждое подтверждённое фото становится новым положительным прототипом карточки. Исправленный вариант даёт:

- positive: `(photo, выбранная карточка)`;
- hard negative: `(photo, ошибочно предложенная карточка)`;
- остальные top-k — слабые negatives.

Быстрый персонализированный score:

```text
feedbackScore =
  max similarity с positive-примерами
- 0.3 * max similarity с hard negatives
```

Добавить его с весом `0.15–0.30`. Это начнёт улучшать выдачу после первых десятков подтверждений, без переобучения.

Важно фильтровать дубли/perceptual hash и ограничивать вклад одного пользователя, иначе один человек сможет перекосить карточку.

5. **После накопления 500–2000 пар — обучить лёгкий reranker, 3–5 дней**

Не fine-tune всей SigLIP на малом датасете. Сначала обучить небольшой MLP/LightGBM на признаках:

- top image cosine, mean top-3;
- text cosine;
- положительные и отрицательные feedback similarities;
- категория, OCR match;
- разрыв между первым и вторым кандидатом;
- число реальных прототипов карточки.

Loss — pairwise ranking: правильная карточка выше сохранённого hard negative. Такой reranker дешёв, работает за миллисекунды и обычно полезнее преждевременного LoRA.

При нескольких тысячах разнообразных примеров можно попробовать LoRA/adapters на vision-проекции с contrastive/triplet loss, обязательно сохранив замороженную baseline-модель для сравнения.

6. **Двухступенчатый поиск и reranking — 2 дня**

- SigLIP извлекает top-30.
- Отсекаются несовместимые типы: напиток/еда/бутылка/меню.
- Fusion + feedback reranker выбирает top-5.
- Опционально запускать маленький food-classifier как дополнительный признак, а не окончательный классификатор. Food-101 плохо покрывает российские блюда и конкретные позиции ресторанов.
- VLM вроде Qwen-VL применять только на неоднозначных top-5 и только если CPU latency укладывается в бюджет. На Railway CPU генеративная VLM, вероятнее всего, станет медленным fallback, а не основным путём.

7. **Калибровка порогов и правильный UX — 1–2 дня**

Не показывать cosine как `%`. Откалибровать confidence на test-наборе через logistic/isotonic regression, используя score и margin:

- автооткрытие — только при calibrated probability ≥ 0.9 и достаточном разрыве top1/top2;
- уверенная подсказка — top-3;
- низкая уверенность — категории/поиск по названию;
- если все кандидаты слабые: «Не уверены, что это за блюдо» вместо случайного результата.

Полезные UX-подсказки:

- «Снимите блюдо сверху, целиком, без сильного размытия».
- Отдельный режим «блюдо / бутылка / меню».
- Показать 3–5 вариантов без процентов: «Наиболее похоже».
- Кнопки «Ничего из этого» и быстрый текстовый поиск.
- «Попробовать ещё раз» повторяет анализ того же фото; «Переснять» открывает камеру.
- После выбора заведения rerank ограничить его меню — это один из сильнейших сигналов точности.

**Рекомендуемый порядок внедрения:** benchmark → SigLIP2 A/B → реальные multi-photo прототипы → text fusion → online feedback kNN → калибровка → лёгкий reranker. Это даст основную прибавку примерно за **2 недели**, не требует дорогого GPU и укладывается в 3–5 секунд после прогрева. MobileCLIP2 можно включить в benchmark как более быстрый вариант, но не выбирать вслепую: его преимущество прежде всего в соотношении скорости и общей zero-shot точности. [MobileCLIP](https://github.com/apple/ml-mobileclip).
